# 具体例: 35m 級の灯台を設計する

`examples/lighthouse-35-immersive.boxel.json` は、岩礁基壇付きの 35m 級灯台を具体例として作った設計図です。
再生成スクリプトは `scripts/generate-lighthouse.js` にあります。

このページでは、完成形そのものよりも「どう考えて詰めたか」を残します。

## 最初に固定したこと

- スケールは推奨規約どおり `1 block = 1 meter`
- 全高は `35m 級`、実際の bounds は `29 x 37 x 34`
- 外観は `岩場 + 円形基壇 + 白い塔身 + 灯室 + 赤い屋根`
- 要件は「外から見て灯台らしい」だけでなく「中に入って上まで登れる」
- 色は先に `palette` を切り、岩、基壇、主壁、影、金物、ガラス、屋根を分ける

最初にこれを固定すると、途中でディテールを盛っても全体が崩れにくくなります。

## 設計の順番

細部から作ると破綻しやすいので、外形から順に詰めます。

1. 岩場と基壇を作る
2. 主塔の大きなボリュームを作る
3. 入口と付属舎を付ける
4. 内部床と階段を通す
5. 灯室とバルコニーを載せる
6. 最後に窓、帯、手すり、灯火などの表情を足す

今回のスクリプトでも、`buildRockBase` → `buildFoundation` → `buildTower` → `buildAnnex` → `buildInterior` → `buildLantern` の順で積んでいます。

## 設計で気を付けたこと

### 1. 塔の見た目と内部空間を同時に成立させる

灯台は外から見ると細く、内側は階段が必要です。
そのため、下部は外半径 6 / 内半径 5、上部は外半径 5 / 内半径 4 の二段構成にしました。

- 下部を少し太くして入口まわりの安定感を出す
- 上部を少し絞って「灯台らしいテーパー感」を出す
- 内径は 1 ブロックにしない
- 中央に完全な柱を作らず、螺旋階段が回る空洞を残す

見た目だけで細くすると、階段を置いた瞬間に内部が詰まります。

### 2. 壁厚は基本 1m、厚く見せたい場所だけ帯で処理する

README の推奨どおり、壁厚は原則 `1 block` を基準にしました。
重厚感は壁そのものを厚くするのではなく、以下で出しています。

- 基壇の笠石リング
- 塔中腹の白い帯
- 窓まわりの明色トリム
- 影側だけ別色に塗る処理

「厚み感を出したい = 壁を何ブロックも厚くする」ではない、というのが重要です。

### 3. 入口から最上部までの導線を先に考える

今回は「中に入れること」が要件だったので、外装より先に次を意識しました。

- 入口の空洞位置
- 付属舎から塔への接続位置
- 踊り場を置く高さ
- 灯室直下でどう折り返すか
- `check-access` でどこを `from` / `to` にするか

`check-access` は空間中の空セル同士を検証するので、床や灯火のブロックを目的地にすると失敗します。
灯室の中心ではなく、実際に人が立つ空点を `to` にする必要があります。

### 4. 円筒だけで終わらせず、付属物でスケール感を出す

灯台は本体だけだと「ただの塔」になりやすいです。
今回は次を足して、用途が読めるようにしました。

- 岩礁基壇
- 付属舎
- 桟橋側の進入路
- バルコニー支柱
- 灯室の金物
- レンズと頂部の灯火

建物の用途が一目で分かる部位を 2 つ以上入れると、完成度がかなり上がります。

## 苦労したことと解決策

### 1. 螺旋階段を置くと、見た目より中が狭い

問題:
- 円塔の内径は数字上十分でも、ボクセルにすると斜め方向が急に詰まる
- 階段 1 段ごとに 1 ブロック上げると、上層で壁や床と干渉しやすい

解決:
- 階段は円周上に 1 点ずつ置くのではなく、4 段ごとに踏面を少し広げた
- 中央床は全面を残さず、吹き抜けを開けた
- 10m / 18m 付近に踊り場を入れて、導線を読みやすくした

### 2. 灯室の下で導線が切れやすい

問題:
- 塔本体の螺旋階段を上まで回しても、灯室の床や手すりの位置で最後の 1 手が塞がりやすい

解決:
- 灯室直下に小さな橋を追加した
- その上にハッチ状の踏面を 2 ブロックだけ置き、灯室入口へつなげた
- `render --y 26` と `check-access` を併用して、上層だけを重点的に確認した

この手の詰まりは、全景の `ortho` だけでは見落としやすいです。

### 3. 白い塔はのっぺり見えやすい

問題:
- 単色の円塔は、形が合っていても模型っぽく見える

解決:
- 影側にだけ `tower-shadow` を部分的に塗る
- 中腹と上部に水平帯を入れる
- 窓とトリムを明暗で分ける

外形を変えなくても、色のレイヤーだけでかなり立体感が出ます。

### 4. `validate --lint` の壁厚警告

問題:
- 岩場と基壇の塊を、厚すぎる壁としてヒューリスティックに拾うことがある

解決:
- まず `ortho` と実寸を見て、本当に壁が厚すぎるのかを確認する
- 今回は塔壁ではなく地形側が原因だったので、そのままにした

lint は重要ですが、地形や基壇まで機械的に「悪い」とは限りません。

## スクリプト化で意識したこと

### 1. 手作業より、再生成可能性を優先する

今回は CLI を何十回も叩くより、`scripts/generate-lighthouse.js` に寄せました。
理由は以下です。

- 円盤、リング、円柱、半球を何度も組み合わせる必要がある
- 微調整で半径や高さを変える回数が多い
- 途中で導線を修正したときに全体を再生成したい

複雑な建築は、最終物だけでなく「どう再現するか」を残した方が後続が触りやすくなります。

### 2. ビルド順を関数で固定する

スクリプトでは、部位ごとに関数を分けて最後に順番に呼んでいます。

- 地形
- 基壇
- 塔
- 付属舎
- 内装
- 灯室
- 外装の仕上げ

こうしておくと、どの工程がどこを上書きしているかを追いやすくなります。

### 3. 「追加」と「削る」を両方持つ

ボクセル建築では、埋める処理だけでは足りません。
今回は `setBlock` だけでなく `clearBox` も用意して、次を作りやすくしました。

- 入口のくり抜き
- 付属舎の内部空洞
- 各階の吹き抜け
- 灯室入口の開口

最初に大きく作って、あとで削る方が形は安定します。

### 4. 色の上書きを前提にする

最初からすべての色を正確に置こうとすると、処理順が過度に複雑になります。
今回はまずボリュームを置き、そのあとで以下を上書きしています。

- 影側の壁色
- 窓枠
- 扉
- 最上部の灯火

ボクセルでは「最後に塗ったものが勝つ」前提で組んだ方が楽です。

### 5. bounds は最後に計算する

部位追加や削除が多いので、bounds は途中で固定しません。
全ブロックを書き終わってから計算しています。

再生成スクリプトで bounds を手書きすると、後でほぼ確実にズレます。

## 検証フロー

仕上げでは、次の順で見ると効率が良いです。

1. `boxel validate <file> --lint`
2. `boxel check <file> --max-x ... --max-y ... --max-z ...`
3. `boxel ortho <file> --crop structure --center --grid 5`
4. `boxel render <file> --y 主要階高 --color`
5. `boxel check-access <file> --from 入口側の空点 --to 上階の空点`

今回の灯台では特に `render --y 26` と `check-access` が有効でした。
上層の一歩分の詰まりは、全景より断面の方が見つけやすいためです。

## 後続向けの流用ポイント

- 塔を別用途へ流用するなら、先に「内径が足りるか」を決める
- 城や時計塔へ応用するなら、螺旋階段と踊り場の考え方はそのまま使える
- 地形付き建築では、lint の壁厚警告は必ず目視で意味を確認する
- 迷ったら、細部より先に「入口から最上部までの導線」を成立させる

完成度を上げる順番は、いつも「外観の豪華さ」ではなく「骨格」「導線」「用途が読める部位」です。
